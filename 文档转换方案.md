# Verse 文档中文化与 PDF 构建方案

## 1. 目标与边界

将本仓库的 Verse 文档完整中文化，并稳定构建为可发布、可复现的 PDF：英文原文版、既有简体中文版，以及基于大语言模型重译的简体中文版。原英文文档和当前英文网站必须继续保留，不以翻译结果覆盖原文；新译稿也不得覆盖既有 `docs/zh/` 译文。

本次范围包括 `docs/` 中的 22 个 Markdown 文件（当前约 837 KB）、MkDocs 的中文导航和页面元数据；代码示例、文件名、API 名称、Verse 关键字及其语义不翻译。图片、SVG、JavaScript 与 CSS 等二进制或程序资源无需翻译，但须能正常进入中文网站和 PDF。

`VerseSyntaxValidation.md` 目前未在网站导航内。若“全部内容”指仓库内所有文档，它也应翻译，并在中文 PDF 中作为附录收录。

## 2. 交付物

当前已落地的构建文件如下：

```text
docs/
  zh/                         # 所有中文 Markdown 页面，与英文文件一一对应
    index.md
    00_overview.md
    ...
    18_evolution.md
    concept_index.md
    VerseSyntaxValidation.md
  zh/术语表.md                  # 供译者和审校使用，不放入正文导航
  zh-llm/                     # 大语言模型重译版 Markdown，与英文文件一一对应
    index.md
    00_overview.md
    ...
    18_evolution.md
    concept_index.md
    VerseSyntaxValidation.md
requirements-pdf.txt           # HTML 组装所需的 Python 依赖
scripts/
  build_pdf.py                 # 合并 22 个 Markdown 页面并生成可打印 HTML
  render_pdf.mjs               # 使用 Chromium/Playwright 打印 A4 PDF
  build_pdfs.ps1               # Windows 一键构建英文、既有中文与 LLM 中文 PDF
styles/
  pdf-book.css                 # 封面、目录、正文、代码块和表格的打印样式
output/pdf/
  Book-of-Verse-en.pdf         # 英文构建产物，默认不提交版本库
  Book-of-Verse-zh.pdf         # 中文构建产物，默认不提交版本库
  Book-of-Verse-zh-llm.pdf     # LLM 重译版中文构建产物，默认不提交版本库
```

英文站点继续使用现有的 `mkdocs.yml` 与 `docs/*.md`。PDF 构建直接以 `docs/*.md`、`docs/zh/*.md` 或 `docs/zh-llm/*.md` 为输入，不改变官方英文网站配置；因此原文更新、译文更新、LLM 重译稿更新与 PDF 发布彼此隔离。

## 3. 翻译策略

### 3.1 文件与链接处理

1. 按现有导航顺序创建中文文件，文件名保持不变。既有人工校对译文放在 `docs/zh/` 下；LLM 重译稿放在 `docs/zh-llm/` 下。
2. 将正文、标题、表格、提示框、图注和导航名称译为简体中文；保留 Markdown 结构、锚点层级和表格格式。
3. 将 Markdown 内部链接改为对应的中文文件，例如 `00_overview.md` 改为 `zh/00_overview.md`（同目录引用则使用正确的相对路径）。外部链接保持原样。
4. 对英文技术名词第一次出现采用“中文（English）”形式，后续统一使用术语表规定的中文或英文写法。
5. 代码围栏、行内代码、Verse 标识符、效果标注（如 `<suspends>`）、类型名、API 名、URL、命令行参数、JSON 键和预期程序输出一律不翻译。代码注释若属于教学说明则译为中文，但不能改动可执行代码。

### 3.2 术语表与一致性

先建立 `docs/zh/术语表.md`，在翻译前锁定高频术语。初始建议如下，最终以 UEFN/Verse 官方中文术语为准：

| 英文 | 建议中文 | 备注 |
| --- | --- | --- |
| Verse | Verse | 语言名称不翻译 |
| expression | 表达式 | |
| failure | 失败 | 与异常（exception）区分 |
| failable expression | 可失败表达式 | |
| effect | 效果 | 保留 effect specifier 的英文代码形式 |
| concurrency | 并发 | |
| persistable | 可持久化 | |
| module | 模块 | |
| interface | 接口 | |
| class | 类 | |
| type | 类型 | |
| mutability | 可变性 | |

每完成一个章节，应做一次术语扫描，避免同一概念出现多个中文译名。

### 3.3 执行批次

为保证质量，按下列批次翻译和审校，而不是一次性机械替换：

1. 基础：主页、概览、表达式、原始类型、容器、运算符。
2. 语言结构：可变性、函数、控制流、失败、结构体与枚举、类与接口。
3. 高级主题：类型系统、访问说明符、效果、并发、实时变量、模块与路径、可持久化类型、演进。
4. 索引与附录：概念索引、语法验证文档。

每批完成后执行构建、链接检查和抽样人工审校；重点检查长文档 `10_classes_interfaces.md`、`11_types.md`、`14_concurrency.md` 与 `16_modules.md`。

## 4. PDF 构建方案

### 4.1 已采用实现

使用 `scripts/build_pdf.py` 将固定导航顺序的 22 个 Markdown 页面分别汇总为英文、既有中文或 LLM 重译中文的单一 HTML，再由 `scripts/render_pdf.mjs` 通过本机 Chromium 打印为 PDF。此链路已在 Windows 上验证，避免了 `mkdocs-with-pdf` 及其排版依赖的兼容性问题。

构建器执行以下工作：

- 以 `index.md`、`00_overview.md` 至 `18_evolution.md`、`concept_index.md`、`VerseSyntaxValidation.md` 的固定顺序组书；
- 在首部生成封面和章节目录，并将 `VerseSyntaxValidation.md` 作为附录；
- 将跨 Markdown 页面链接改写为 PDF 内的章节链接；
- 去除仅供 MkDocs 使用的 YAML front matter；
- 使用 A4、页眉、页码、目录、可换行代码块和可分页表格；
- 将英文输出为 `output/pdf/Book-of-Verse-en.pdf`，将既有中文输出为 `output/pdf/Book-of-Verse-zh.pdf`，将 LLM 重译版输出为 `output/pdf/Book-of-Verse-zh-llm.pdf`。

`build_pdf.py` 当前支持三个 edition：

| edition | 输入目录 | 输出 PDF |
| --- | --- | --- |
| `en` | `docs/` | `output/pdf/Book-of-Verse-en.pdf` |
| `zh` | `docs/zh/` | `output/pdf/Book-of-Verse-zh.pdf` |
| `zh-llm` | `docs/zh-llm/` | `output/pdf/Book-of-Verse-zh-llm.pdf` |

`zh-llm` edition 会复用与既有中文 PDF 相同的封面、目录、字体和打印样式。构建器会在读取 Markdown 时移除文件开头的 UTF-8 BOM，避免带 BOM 的译文标题无法被 Markdown 正确识别。

Chromium 路径可通过环境变量 `PDF_CHROME_PATH` 覆盖。构建脚本优先使用系统 Node.js 和已安装的 Playwright；在 Codex 桌面环境中会自动复用其运行时。若在普通开发机中首次构建且未安装 Playwright，脚本会提示或通过 npm 安装该依赖，但仍使用系统 Chrome/Edge，不下载额外浏览器。

### 4.2 中文字体

PDF 成败的关键是可嵌入的中文字体。`styles/pdf-book.css` 已按优先级指定 Noto/思源 CJK 字体、微软雅黑及系统后备字体，代码块优先使用 Cascadia Mono、Noto Sans Mono CJK SC 或 Consolas。每次在新的构建机上仍应检查：

- 标题、正文、表格及目录没有方框或乱码；
- 代码块中的英文符号与缩进正常；
- PDF 在未安装开发机字体的另一台设备上仍可阅读。

## 5. 自动校验与人工验收

构建与验收应至少验证以下项目：

1. `build_pdf.py` 会验证所选中文输入目录与英文 Markdown 清单完全一致，且组书顺序覆盖全部 22 个源文件；中文术语表不进入正文。
2. 每次翻译更新后，比对修改文件的代码围栏数量、语言标记、HTML 注释和 Verse 代码；可执行代码、标识符、效果标注、URL 与预期输出不得变更。
3. 检查修改页面的内部 Markdown 链接与图片引用；PDF 内部跨章链接应能跳转到对应章节。
4. 所选 PDF 均存在、页数大于 0，且能通过 `pdfinfo`、`pypdf` 或等效工具读取。
5. 渲染并检查封面、目录、每个改动章节的首尾页、长代码块、表格、概念索引和附录；不得出现文字重叠、裁切、乱码或空白页。
6. 用文本提取工具抽检中文 PDF，确认其中含中文正文而非只有图片。

人工语言审校至少覆盖：术语一致性、否定与条件语义、失败/效果/并发等易误译概念，以及代码输出与说明是否仍匹配。

## 6. 建议命令与发布流程

日常双语构建统一由 PowerShell 脚本封装：

```powershell
.\scripts\build_pdfs.ps1
```

也可只输出一个语言版本：

```powershell
.\scripts\build_pdfs.ps1 -Language en
.\scripts\build_pdfs.ps1 -Language zh
.\scripts\build_pdfs.ps1 -Language zh-llm
```

脚本会创建或复用 `.venv`、安装 `requirements-pdf.txt`、检查所选版本的文件清单、生成临时 HTML，并输出 PDF 至 `output/pdf/`。临时 HTML 位于 `tmp/pdfs/`；PDF 构建产物均已被 `.gitignore` 忽略。CI 可复用同一命令并上传需要发布的 PDF 作为构建产物。

### 6.1 LLM 重译版 PDF 流程

当 `docs/zh-llm/` 中 22 个 Markdown 文件已经由大语言模型翻译完成后，使用以下命令生成重译版 PDF：

```powershell
.\scripts\build_pdfs.ps1 -Language zh-llm
```

输出文件为：

```text
output/pdf/Book-of-Verse-zh-llm.pdf
```

推荐验收步骤：

1. 确认 `docs/zh-llm/` 中存在与 `docs/*.md` 一一对应的 22 个 Markdown 文件。
2. 运行 `.\scripts\build_pdfs.ps1 -Language zh-llm`。
3. 用 `pdfinfo` 或等效工具确认 PDF 页数大于 0、页面尺寸为 A4、文件未损坏。
4. 渲染抽检封面、目录、正文首页、长代码块和附录页面，确认中文字体、页眉页脚和目录标题正常。
5. 若目录标题异常显示为代码注释或英文隐藏测试内容，优先检查译文文件开头是否带 BOM；当前构建器已自动移除开头 BOM。

### 6.2 上游文档更新后的增量流程

后续拉取上游 Git 更新后，不应直接用机器翻译覆盖已审校的 `docs/zh/` 文件。推荐按以下顺序处理：

1. 查看英文文档变更范围，例如 `git diff <上次同步提交>..HEAD -- docs/`，并仅列出新增、修改或删除的 Markdown 页面。
2. 对新增英文页面，在 `docs/zh/` 或 `docs/zh-llm/` 创建同名文件；对修改页面，只翻译受影响的段落、表格、提示框和教学性代码注释。`scripts/translate_docs_zh.py` 可用于创建既有机翻初稿，但不得以 `--overwrite` 覆盖已校对译文；LLM 重译稿应继续放在 `docs/zh-llm/` 中。
3. 以 `docs/zh/术语表.md` 为准进行术语校对，并重新核对代码围栏、Verse 代码、HTML 注释、链接和锚点。
4. 若新增或删除了英文 Markdown 文件，同步更新 `BOOK_FILES`（`scripts/build_pdf.py`）中的组书顺序；每个英文文件必须有同名中文译文。
5. 运行 `./scripts/build_pdfs.ps1` 或 `./scripts/build_pdfs.ps1 -Language zh-llm`，完成本节的自动检查与视觉抽检后再提交译文与构建脚本变更。

这样，每次上游更新都能在保留既有人工校对成果的前提下，重新交付内容一致的英文、既有中文和 LLM 重译版中文 PDF。

## 7. 实施顺序与完成标准

1. 已完成：建立 22 个中文 Markdown 文件、术语表、双语 PDF 构建脚本和打印样式，并验证中文字体、目录、代码块和附录。
2. 已完成：生成英文、既有中文与 LLM 重译版中文 PDF，并完成封面、目录、正文、长代码块和附录的渲染抽检。
3. 待持续执行：每次上游文档更新时按 6.2 完成增量翻译、术语与结构审校，再重新构建所需 PDF。
4. 可选后续工作：在 CI 中运行 `build_pdfs.ps1` 并上传需要发布的 PDF；本地双语网站构建见根目录 `网站实施方案.md`，不改变本 PDF 流程。

完成标准为：英文网站仍能按原方式构建；所选中文输入目录中的 22 个 Markdown 文档全部存在且与英文一一对应；一条命令能稳定输出无乱码、带封面、目录和页码的完整英文、既有中文或 LLM 重译版中文 PDF；每次上游更新只需增量翻译变更内容并重复该构建与验收流程。
