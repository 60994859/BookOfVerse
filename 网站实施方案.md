# 网站实施方案

## 1. 目标

在本机生成一个与官方 `https://verselang.github.io/book/` 类似的 MkDocs Material 网站，同时支持英文原文和 `docs/zh-llm/` 中的大语言模型中文译文。

本方案不修改官方 `mkdocs.yml`，也不覆盖 `docs/zh/` 既有中文译文。网站构建使用单独配置文件和临时构建目录，便于随时重建和删除。

## 2. 站点形态

本地站点采用双语静态站结构：

```text
site/
  index.html      # 入口页，默认跳转到中文站
  zh/             # 简体中文站点，来自 docs/zh-llm/
  en/             # 英文站点，来自 docs/*.md
```

访问地址：

```text
http://127.0.0.1:8000/zh/   # 中文站
http://127.0.0.1:8000/en/   # 英文站
http://127.0.0.1:8000/      # 入口页，默认跳转中文站
```

语言切换使用 MkDocs Material 的 `extra.alternate` 配置。页面顶部会出现 English / 简体中文的语言切换入口。

## 3. 新增文件

本次新增或使用的站点构建文件如下：

```text
mkdocs.local.en.yml          # 本地英文站点配置
mkdocs.local.zh-llm.yml      # 本地中文 LLM 译文站点配置
scripts/build_local_site.ps1 # 一键构建本地双语站点
```

构建脚本会生成以下临时或输出目录：

```text
tmp/site-src/                # 临时 MkDocs 源目录，不提交版本库
site/                        # 静态站点输出目录，不提交版本库
```

`.gitignore` 已忽略：

```text
site/
tmp/site-src/
```

## 4. 输入来源

英文站点输入：

```text
docs/*.md
docs/extra.css
docs/images/
docs/js/
docs/Assets/
```

中文站点输入：

```text
docs/zh-llm/*.md
docs/extra.css
docs/images/
docs/js/
docs/Assets/
```

构建时不会直接把 `docs/zh-llm/` 作为 MkDocs `docs_dir`，而是先复制到临时目录 `tmp/site-src/zh/`。这样可以在临时目录中做网站所需的兼容处理，而不改动原始译文文件。

## 5. 构建流程

运行：

```powershell
.\scripts\build_local_site.ps1
```

脚本执行以下步骤：

1. 创建或复用 `.venv`。
2. 安装 `requirements.txt` 中的 MkDocs、Material 主题和插件依赖。
3. 清空并重建 `tmp/site-src/en/` 与 `tmp/site-src/zh/`。
4. 将 `docs/*.md` 复制到 `tmp/site-src/en/`。
5. 将 `docs/zh-llm/*.md` 复制到 `tmp/site-src/zh/`。
6. 将共享资源 `extra.css`、`images/`、`js/`、`Assets/` 复制到两个临时源目录。
7. 移除临时 Markdown 文件开头的 UTF-8 BOM。
8. 为中文站临时文件补充英文原文锚点，保证原有 `#anchor` 链接仍可跳转。
9. 使用 `mkdocs.local.en.yml` 构建英文站到 `site/en/`。
10. 使用 `mkdocs.local.zh-llm.yml` 构建中文站到 `site/zh/`。
11. 生成 `site/index.html`，默认跳转到 `site/zh/`。

## 6. 本地预览

构建后可以直接启动本地静态 HTTP 服务：

```powershell
.\scripts\build_local_site.ps1 -Serve
```

默认端口是 `8000`。如果端口被占用，可以指定其他端口：

```powershell
.\scripts\build_local_site.ps1 -Serve -Port 8001
```

也可以在已经生成 `site/` 后手动启动服务：

```powershell
.\.venv\Scripts\python.exe -m http.server 8000 --bind 127.0.0.1 --directory site
```

然后访问：

```text
http://127.0.0.1:8000/zh/
http://127.0.0.1:8000/en/
```

建议通过 HTTP 服务访问，而不是直接双击 HTML 文件。MkDocs Material 的搜索、资源路径和语言切换在 HTTP 服务下更稳定。

## 7. 中文锚点兼容

英文文档中的内部链接经常指向英文标题生成的锚点，例如：

```text
11_types.md#understanding-subtyping
10_classes_interfaces.md#using-castable_subtype
```

中文翻译后标题会变成中文，MkDocs 默认生成的锚点也会改变，导致这些链接失效。

`scripts/build_local_site.ps1` 会在临时目录中执行兼容处理：

1. 读取英文源文件标题，计算原英文标题对应的 MkDocs 锚点。
2. 读取中文译文中相同位置的标题。
3. 在 `tmp/site-src/zh/*.md` 的中文标题后追加原英文锚点属性。
4. 对 `castable_subtype`、`classifiable_subset` 等带下划线的术语标题，保留带下划线的兼容锚点。

这个处理只发生在 `tmp/site-src/zh/`，不会改动 `docs/zh-llm/`。

## 8. 配置说明

`mkdocs.local.en.yml` 与官方 `mkdocs.yml` 保持相同主题风格，主要差异是：

1. `docs_dir` 指向 `tmp/site-src/en`。
2. `site_dir` 指向 `site/en`。
3. 增加语言切换配置。
4. 去掉依赖 Git 历史的 revision date 插件，减少本地构建不确定性。
5. 将 `VerseSyntaxValidation.md` 作为附录放入导航。

`mkdocs.local.zh-llm.yml` 主要差异是：

1. `docs_dir` 指向 `tmp/site-src/zh`。
2. `site_dir` 指向 `site/zh`。
3. 站点名称、描述和导航文字使用简体中文。
4. 使用同一套 `extra.css`、`images/`、`js/` 与 `Assets/` 资源。
5. 使用同一套语言切换配置。

## 9. 代码高亮

官方网站使用 `docs/js/Verse.js` 加载 Shiki，并读取 `docs/Assets/Verse.json`、`VerseLight.json`、`VerseDark.json` 来高亮 `language-verse` 代码块。

本地双语站点复用同一套机制：

```text
site/zh/js/Verse.js
site/zh/Assets/Verse.json
site/zh/Assets/VerseLight.json
site/zh/Assets/VerseDark.json
site/en/js/Verse.js
site/en/Assets/Verse.json
site/en/Assets/VerseLight.json
site/en/Assets/VerseDark.json
```

`Verse.js` 不再写死官方站点的 `/book/Assets/` 路径，而是通过 `import.meta.url` 按当前脚本位置推导资源目录。因此同一脚本可同时适配：

```text
https://verselang.github.io/book/
http://127.0.0.1:8000/zh/
http://127.0.0.1:8000/en/
```

注意：Shiki 模块从 `https://esm.sh/shiki@3.0.0` 加载。本地预览时如果浏览器无法访问外网，代码块仍会显示为普通代码块，但不会有 Verse 语法高亮。

## 10. 验收项目

每次构建后至少检查：

1. `site/zh/index.html` 存在。
2. `site/en/index.html` 存在。
3. `http://127.0.0.1:8000/zh/` 能打开中文首页。
4. `http://127.0.0.1:8000/en/` 能打开英文首页。
5. 中文站导航包含 22 个文档页面，其中 `VerseSyntaxValidation.md` 作为附录。
6. 顶部语言切换入口能在中文和英文之间切换。
7. 搜索框能正常加载。
8. 随机打开 `08_failure.md`、`10_classes_interfaces.md`、`11_types.md`、`14_concurrency.md`，确认正文、代码块和内部链接正常。
9. 打开包含 Verse 代码块的页面，确认代码块有不同颜色的语法高亮。
10. 构建输出中没有中文文档内部链接警告。

MkDocs Material 可能会输出关于未来 MkDocs 2.0 的提示信息。这是主题自身的版本提示，不代表当前站点构建失败。

## 11. GitHub Pages 自动部署

当前仓库已经配置 GitHub Actions，将本地双语站点部署到 GitHub Pages。

部署 workflow：

```text
.github/workflows/deploy.yml
```

触发方式：

1. 每次 push 到 `main` 分支自动触发。
2. 也可以在 GitHub 网页的 **Actions** 页面手动运行。

部署流程：

1. Checkout 仓库。
2. 安装 Python 3.11。
3. 配置 GitHub Pages。
4. 执行：

```powershell
./scripts/build_local_site.ps1 -Python python
```

5. 上传生成的 `site/` 目录作为 Pages artifact。
6. 使用 `actions/deploy-pages@v4` 发布到 GitHub Pages。

GitHub Pages 访问地址预计为：

```text
https://60994859.github.io/BookOfVerse/
https://60994859.github.io/BookOfVerse/zh/
https://60994859.github.io/BookOfVerse/en/
```

由于这是 GitHub 项目页，站点基础路径是 `/BookOfVerse/`。workflow 会在构建时设置：

```text
SITE_URL_EN=https://60994859.github.io/BookOfVerse/en/
SITE_URL_ZH=https://60994859.github.io/BookOfVerse/zh/
SITE_ALTERNATE_EN=/BookOfVerse/en/
SITE_ALTERNATE_ZH=/BookOfVerse/zh/
```

这样部署后的语言切换链接会指向正确的 GitHub Pages 路径。

如果 GitHub Actions 的 `deploy` 阶段失败，请在 GitHub 仓库网页确认：

```text
Settings -> Pages -> Build and deployment -> Source -> GitHub Actions
```

同时确认：

```text
Settings -> Actions -> General -> Workflow permissions
```

建议允许 workflow 具备发布 Pages 所需权限。当前 workflow 已声明：

```yaml
permissions:
  contents: read
  pages: write
  id-token: write
```

## 12. 后续发布思路

当前方案用于本地预览和静态产物生成。如果后续要发布到 GitHub Pages 或其他静态站点服务，可以直接发布 `site/` 目录内容。

推荐发布结构：

```text
/
  index.html
  zh/
  en/
```

如果要替换官方英文站点之外的独立中文站，可只发布 `site/zh/`。如果要保留中英文切换，则应同时发布 `site/en/` 与 `site/zh/`。
