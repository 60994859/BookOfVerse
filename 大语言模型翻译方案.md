# 大语言模型翻译方案

## 1. 目标

使用低成本大语言模型重新翻译本仓库的 Verse 英文文档，产出一套新的简体中文 Markdown 译稿。新译稿必须保持英文原文的技术语义、Markdown 结构、代码示例和链接结构，不覆盖当前已经存在的 `docs/zh/` 译文。

本方案面向执行翻译的大语言模型使用。模型应把自己当作“技术文档译者 + Markdown 保真编辑器”，而不是摘要工具。所有原文内容都要完整翻译，不能删减、改写为概述或额外添加解释。

## 2. 输出目录

不要修改以下既有目录：

```text
docs/zh/
```

新的 LLM 译稿统一放在独立目录：

```text
docs/zh-llm/
```

目标目录建议结构如下：

```text
docs/zh-llm/
  index.md
  00_overview.md
  01_expressions.md
  ...
  18_evolution.md
  concept_index.md
  VerseSyntaxValidation.md
  术语表.md
```

`docs/zh-llm/术语表.md` 可以先从 `docs/zh/术语表.md` 复制而来。翻译过程中如发现必须新增的固定译名，应追加到新术语表中，但不要改动旧术语表。

## 4. 是否一个会话翻译一个 Markdown 文件

建议采用“一个会话翻译一个 Markdown 文件”的方式。

这样做通常比一次性把多个文件塞进同一个会话更好，原因如下：

1. 每个文件拥有相对完整的局部上下文，模型更容易保持本章术语、指代和语气一致。
2. 失败成本低。某个文件翻译不满意时，只需要重做该文件，不影响其他文件。
3. 便于人工审校和版本管理。每个会话对应一个源文件和一个目标文件。
4. 便宜模型的长上下文稳定性通常不如昂贵模型，把任务拆成文件级更可控。

但对较长文件，不建议强行一次输出完整译文。以下文件建议在同一个会话内按二级标题或三级标题分块翻译，再合并为同一个目标文件：

```text
06_functions.md
10_classes_interfaces.md
11_types.md
14_concurrency.md
16_modules.md
```

尤其是 `10_classes_interfaces.md` 较长，应优先按大标题分块。分块时仍保持“一个源文件一个会话”，不要跨文件混用上下文。

## 5. 推荐执行顺序

先做小样本试译，再全量翻译。

第一批试译文件：

```text
00_overview.md
08_failure.md
13_effects.md
14_concurrency.md
```

这几个文件能覆盖概览、失败语义、效果标注和并发等易误译主题。用候选模型各翻一遍后，人工比较以下指标：

1. 是否严格保留 Markdown、代码块、链接目标和表格结构。
2. 是否遵守术语表。
3. 中文是否像技术文档，而不是生硬机翻。
4. 是否准确处理 failure、effect、concurrency、transaction、suspension 等关键概念。
5. 是否擅自增删内容。

试译通过后，再按以下顺序翻译：

```text
index.md

01_expressions.md
02_primitives.md
03_containers.md

```

## 6. 术语表

翻译必须优先遵守 `docs/zh/术语表.md`。当前术语表如下：

| failable expression | 可失败表达式 | |
| effect | 效果 | 保留 effect specifier 的英文代码形式 |
| concurrency | 并发 | |
| persistable | 可持久化 | |
| module | 模块 | |

| type | 类型 | |

| extension field | 扩展字段 | |

额外约定：

1. `Verse`、API 名、类型名、函数名、模块路径、文件名、命令、URL、JSON 键、效果标注和代码标识符不翻译。
2. `<transacts>`、`<suspends>`、`<decides>`、`<computes>`、`<converges>` 等尖括号形式必须原样保留。
3. `failure` 一律译为“失败”，不要译为“故障”“失效”或“异常”。
4. `effect` 在语言机制语境中译为“效果”，不要译为“副作用”。
5. `logic` 在 Verse 类型语境中译为“逻辑”，不要擅自改成 `bool` 或“布尔值”。
6. 首次出现的核心术语可以使用“中文（English）”形式，但不要每次重复英文。

## 7. Markdown 保真规则

翻译时必须保持 Markdown 文件仍可被 MkDocs、PDF 构建脚本和普通 Markdown 工具处理。

必须原样保留：

1. 文件名。
2. Markdown 标题层级，也就是 `#`、`##`、`###` 的数量。
3. 代码围栏标记，例如 ```` ```verse ````、```` ```text ````。
4. 代码块内的代码内容。第一轮翻译建议完整保留 fenced code block，不翻译其中注释，避免破坏可执行示例。
5. 行内代码，例如 `` `if` ``、`` `array` ``、`` `Foo()` ``。
6. HTML 标签、HTML 注释和锚点。
7. 图片路径、链接目标、URL、邮箱地址。
8. 表格列数、分隔行、单元格顺序。
9. MkDocs admonition 标记，例如 `!!! note`、`??? example`。

可以翻译：

1. 普通正文。
2. 标题文字，但保留标题级别。
3. 表格中的自然语言说明。
4. 链接显示文字，但不要改链接目标。
5. 提示框、注释框、图注中的自然语言。

关于链接：

1. 链接文字可以翻译。
2. 链接目标必须原样保留，例如 `[Overview](00_overview.md)` 中的 `00_overview.md` 不要改。
3. 目标目录 `docs/zh-llm/` 与英文源文件使用同名文件，因此大多数相对链接保持不变即可在新目录中工作。
4. 如果链接目标包含 `#anchor`，不要凭空改写锚点。标题翻译后可能导致自动锚点变化，这类问题放到后续链接校验步骤处理。

## 8. 中文风格要求

译文应当是自然、准确、克制的中文技术文档。

风格要求：

1. 使用简体中文。
2. 避免翻译腔，优先使用中文技术文档常见表达。
3. 不要为了流畅而改变技术含义。
4. 不要擅自补充原文没有的解释。
5. 不要省略原文中的限制条件、否定、例外和前提。
6. 对 “must”、“should”、“may”、“can” 等情态动词要区分强度。
7. 长句可以拆成较自然的中文句子，但不能丢失逻辑关系。
8. 对列表项保持原有并列结构，避免把列表改写成段落。

## 9. 单文件翻译提示词模板

每次新开一个会话时，把下面模板发给模型，然后附上待翻译的完整 Markdown 文件内容。

```text

1. 只输出翻译后的 Markdown 正文，不要输出解释、总结、前言或代码块外包装。
2. 不要省略、扩写或重排原文内容。
3. 保留 Markdown 结构、标题层级、表格结构、列表结构和 admonition 标记。
4. 保留所有代码块、行内代码、API 名、类型名、函数名、模块路径、文件名、URL、链接目标、HTML 标签和锚点。
5. 链接文字可以翻译，链接目标必须原样保留。
6. 第一轮翻译中，fenced code block 内的内容必须原样保留，包括注释；不要翻译代码块内注释。
7. 使用自然、准确、克制的简体中文技术文档风格。
8. 严格遵守术语表。
```

## 10. 长文件分块提示词模板

长文件应在同一个会话内分块翻译。每一块都应按原文标题边界切分，不要从段落中间切断。

第一条消息使用以下模板：

```text
你是 Verse 编程语言文档的专业中文译者。本会话只翻译一个 Markdown 文件：<文件名>。

我会按标题边界分多次发送这个文件。请逐块翻译为简体中文。

全局要求：
1. 每次只输出当前分块的译文 Markdown，不要输出解释。
2. 保留 Markdown 结构、标题层级、表格、列表、代码块、行内代码、链接目标、URL、HTML 标签和锚点。
3. fenced code block 内的内容必须原样保留，不翻译其中注释。
4. 严格遵守术语表。
5. 不要重排、摘要、增删内容。
6. 所有分块最终会拼接成 docs/zh-llm/<文件名>，所以每块开头和结尾不要添加额外说明。

术语表：
<粘贴第 6 节术语表>

请等待我发送第 1 块。
```

每个分块使用以下模板：

```text
这是 <文件名> 的第 <N>/<M> 块。请只输出这一块的中文 Markdown 译文。

<粘贴该分块 Markdown 原文>
```

分块建议：

1. 优先按 `##` 标题切分。
2. 如果某个 `##` 仍然太长，再按 `###` 切分。
3. 不要把一个代码围栏拆到两个请求里。
4. 每块完成后，先保存该块译文，再发送下一块。
5. 拼接时保持原始块顺序，不要额外添加分隔符。

## 11. 文件级验收清单

每完成一个文件，都要检查：

1. 目标文件是否保存到 `docs/zh-llm/<同名文件>`。
2. 是否没有修改 `docs/zh/` 中的旧译文。
3. 文件是否仍是合法 UTF-8 Markdown。
4. 代码围栏数量是否与英文源文件一致。
5. 行内代码、URL、链接目标、图片路径是否未被翻译或破坏。
6. 表格列数是否与原文一致。
7. 是否存在明显未翻译的大段英文正文。
8. 是否遵守术语表，尤其是 failure、effect、logic、concurrency、persistable。
9. 是否没有把 `Verse` 翻译成“诗句”“诗歌”等普通词义。
10. 是否没有把代码示例中的标识符、类型名、函数名改成中文。

## 12. 全量验收清单

22 个文件全部完成后，检查：

1. `docs/zh-llm/` 中是否有 22 个与英文源文件同名的 Markdown 文件。
2. 是否另有 `docs/zh-llm/术语表.md`。
3. 英文 `docs/*.md` 与中文 `docs/zh-llm/*.md` 是否一一对应。
4. 所有文件是否能被 Markdown 工具正常打开。
5. 随机抽检至少 4 个重点文件：
   - `08_failure.md`
   - `10_classes_interfaces.md`
   - `13_effects.md`
   - `14_concurrency.md`
6. 抽检长代码块、长表格、内部链接和 admonition。
7. 抽检中文语义是否忠于原文，尤其是条件、否定、因果和限制语。

当前 PDF 构建脚本默认使用 `docs/zh/`。在确认 `docs/zh-llm/` 的质量之前，不要让它替换旧译文。后续如果要基于新译文生成 PDF，可以再单独改造构建脚本，使中文输入目录可配置，例如增加 `--zh-dir docs/zh-llm`，或者在确认后再手动迁移。

## 13. 建议的质量策略

推荐至少执行两轮模型处理：

第一轮：翻译。

目标是生成结构完整、术语基本正确、代码未损坏的中文 Markdown。

第二轮：审校。

把英文原文和中文译文同时提供给模型，让它只检查问题，不重写全文。审校重点：

1. 是否漏译。
2. 是否误译关键技术概念。
3. 是否违反术语表。
4. 是否改坏代码、链接或 Markdown。
5. 是否存在明显不自然的中文句子。

审校提示词可使用：

```text
请对比下面的英文原文和中文译文。只列出需要修改的问题，不要重写全文。

检查重点：
1. 漏译或多译。
2. 技术语义错误。
3. 术语表不一致。
4. Markdown、代码块、行内代码、链接目标、URL 或表格结构被破坏。
5. 中文表达明显不自然。

术语表：
<粘贴第 6 节术语表>

英文原文：
<粘贴原文>

中文译文：
<粘贴译文>
```

## 14. 完成标准

本轮 LLM 重译完成的标准是：

1. `docs/zh-llm/` 中存在完整的 22 个中文 Markdown 文件。
2. `docs/zh/` 旧译文未被覆盖或改动。
3. 所有译文保留原 Markdown 结构和代码示例。
4. 核心术语与 `docs/zh/术语表.md` 一致。
5. 重点章节经人工或模型审校，没有明显机翻腔、漏译或技术误译。
6. 新译文可以作为后续 PDF 构建或中文网站构建的输入候选。
