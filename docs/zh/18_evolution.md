# Verse Code 的演变和兼容性

Verse 采用独特的代码演化方法，其设计的雄心勃勃的目标是创建可以在数十年甚至数百年中保持功能和价值的软件。这一愿景源于 Verse 作为持久性全球元宇宙的编程语言的角色，其中代码必须在广阔的时间尺度上共存、发展并保持兼容性。

Verse 的核心包含塑造代码演变方式的三个基本原则：避免植根于其他语言过去工件的面向未来的设计、代码持久性和兼容性至关重要的元节优先方法以及在编译时捕获运行时问题的强大静态验证。这些原则为一种可以成长和适应的语言奠定了基础，同时保持全球持久代码库所需的稳定性。

## 代码发布的性质

当开发人员将代码发布到 Verse Metaverse 时，他们会与该代码的所有未来用户签订社会契约。这个契约不仅仅是一个约定——它是由语言本身强制执行的。考虑一下当您发布一个简单值时会发生什么：

<!--versetest-->
<!-- 01 -->
```verse
Thing<public>:int = 777
```
这个看似简单的宣言却蕴含着深远的意义。通过将 `Thing` 标记为公开，您就做出了无限期延续到未来的承诺。用户可以相信 `Thing` 始终存在并且始终是一个整数。虽然您保留更改其实际值的自由，但 `Thing` 的存在和类型将成为虚拟宇宙景观中的永久固定装置。

这种持久性超越了简单的值，涵盖了已发布代码的整个结构。持久结构一旦发布到孤岛，就会成为无法更改的不可变模式。封闭枚举永远保持封闭状态，发布后无法接受新值。当类或接口标记有 `<castable>` 属性时，该决定变得不可逆转，因为更改它可能会引入破坏现有代码的不安全转换行为。

发布模型区分两种环境：实时虚拟宇宙和岛屿。在设想的实时元节中，发布尝试更改不可变变量值的更新不会产生任何效果 - 该变量已经存在及其原始值。然而，在当前基于岛屿的实现中，岛屿的新实例将采用更新后的值，提供实用的迁移路径，同时保持概念一致性。

## 向后兼容的架构

Verse 中的向后兼容性不仅仅是简单的语法保留，它还包含有关代码行为方式的语义保证。该语言通过多种机制强制执行这些保证，这些机制协同工作以创建强大的兼容性框架。

功能效果体现了这种方法。当发布具有特定效果（例如 `<reads>`）的函数时，表明它可以读取可变堆数据，这将成为其合同的一部分。该功能的未来版本可能会产生更少的影响（从 `<reads>` 发展到 `<computes>`），但绝不会更多。此限制可确保取决于函数效果配置文件的代码继续正常工作，因为函数只会变得更纯净，而绝不会更少。

类型进化遵循类似的原则。随着时间的推移，类型可能会变得更加具体，例如从 `rational` 更改为 `int`，因为这代表了改进而不是根本性变化。尽管可以添加新字段，但结构必须维护所有现有字段。标有 `<final_super>` 的类永久提交其继承层次结构，确保依赖于特定继承关系的代码仍然有效。

这些规则的执行发生在发布时，而不仅仅是在编译时。 Verse 主动阻止开发人员发布违反兼容性保证的更新，将其他系统中可能出现的运行时失败转变为发布时错误，必须在部署代码之前解决这些错误。

## 管理重大变更

尽管非常强调兼容性，但 Verse 认识到偶尔需要进行一些重大更改。该语言提供了两种管理此类更改的机制：用于逐步迁移的弃用系统和用于重大重大更改的特权。

弃用系统作为一个多阶段过程运行，为开发人员提供了充足的时间来适应。当代码模式被弃用时，它们首先生成警告而不是错误。保存代码时会出现这些警告，提醒开发人员注意未来版本中不支持的做法。代码继续编译和运行，允许项目在制定迁移计划的同时运行。只有当开发人员明确升级到新的语言版本时，弃用才会成为错误，即使如此，保留旧版本的选项也提供了逃生通道。版本 1 引入了几个重要的弃用内容来说明此过程。禁止集合表达式中的失败，以前允许警告，现在需要显式处理可失败表达式。混合分隔符语法会创建隐式块和令人困惑的范围规则，现在必须使用一致的分隔。本地限定符的引入提供了一种消除标识符歧义的新工具，同时不推荐使用“local”作为常规标识符名称。

对于真正的特殊情况，Epic Games 和其他潜在的未来权威机构保留“超能力”，可以在正常兼容性框架之外进行突破性更改。这些权力包括删除已发布实体、以非向后兼容方式更改类型以及出于法律或安全原因重写模块的能力。这些功能承认，成为元节命名空间的好管理者有时需要违反通常的兼容性规则，尽管这种行为应该很少见，并且有令人信服的理由证明是合理的。

## 兼容性规则目录

当您在 Verse 中发布代码时，定义的许多方面都会成为永久的承诺。准确理解什么可以改变、什么不能改变对于设计可以优雅发展的 API 至关重要。该目录记录了 Verse 禁止的更改以及为确保向后兼容性而允许的更改。

这些规则遵循一个一般原则：通常允许使类型更具体（缩小范围）、添加新功能或放宽限制的更改，而通常禁止使类型更通用（扩大）、删除功能或施加新限制的更改。然而，最终/非实例成员和非最终实例成员之间的规则差异很大，后者有更严格的要求。

### 定义

**无法删除公共定义。** 公共变量、函数、类或其他定义发布后，它必须保持可用。删除它会破坏依赖它的任何代码。

**无法更改定义的类型。** 类不能成为结构、接口、枚举或函数。函数不能成为类或类型别名。这些根本性的变化改变了代码与定义的交互方式。

**无法重命名定义。** 重命名相当于删除加添加，这会破坏现有引用。

### 枚举

**无法在封闭枚举中添加或删除枚举器。**封闭枚举（默认）永远提交一组固定的值。代码可以在没有通配符的情况下彻底匹配所有情况，因此添加情况会破坏此类匹配。

**封闭枚举无法变为开放。** 发布为封闭的枚举无法变为开放。这会影响是否可以进行穷举匹配。

**无法对枚举器重新排序。** 枚举值的顺序是公共合约的一部分。

**无法重命名枚举数。** 每个枚举数名称都是永久标识符。

**开放枚举可以添加新的枚举器。**这是枚举的唯一演化路径——开放枚举用详尽匹配的保证换取了增长的能力。

### 类和继承

**课程结束：**

- **可以将非最终类设为最终类。** 添加 `<final>` 说明符可防止将来的继承，这是加强保证的安全添加。
- **不能使最终类成为非最终类。** 一旦类被标记为 `<final>`，删除此限制将允许出现意外的子类，从而可能破坏现有代码中的假设。

**类别唯一性：**

- **可以使非唯一类变得唯一。** 添加 `<unique>` 说明符可启用基于身份的相等性，这不会破坏现有代码。
- **无法使唯一类变得不唯一。** 删除 `<unique>` 会将相等语义从标识更改为未定义，从而破坏依赖于标识比较的代码。

**类抽象性：**

- **可以使抽象类成为非抽象类。** 允许实例化以前的抽象类是一种安全的功能扩展。
- **无法使非抽象类抽象。** 防止先前具体类的实例化会破坏创建实例的代码。**类具体性：**

- **可以使非具体的最终类具体化。**允许最终类的默认实例化是安全的，因为不存在受影响的子类。
- **在其他情况下无法更改具体性。** 将具体类设置为非具体类或更改非最终类的具体性可能会破坏实例化代码或子类行为。

**继承变更：**

- **可以向非抽象类添加继承。** 添加父类或接口可扩展功能，而不会破坏现有功能。
- **无法删除或更改非抽象类的继承。** 删除父级会破坏依赖于继承关系的代码。
- **无法添加、删除或更改抽象类的继承。** 抽象类层次结构必须保持固定，以防止冲突并维护子类型关系。
- **无法添加、删除或更改接口上的继承。**出于同样的原因，接口层次结构必须保持稳定。

**特殊属性：**

- **无法添加或删除 `<castable>` 属性。** 运行时类型检查取决于此属性。在发布后添加它会启用以前不安全的强制转换，而删除它会破坏现有的强制转换。
- **添加后无法删除 `<final_super>`。** 标有 `<final_super>` 的派生类型必须继续从同一父级继承，以维护 `GetCastableFinalSuperClass` 所依赖的类型层次结构。
- **具有 `<final_super>` 的派生类型必须保持从同一父级派生。** 更改父类型会破坏运行时类型查询。

**特殊转换：**

- **可以更改最终类而不继承结构。**这是一个安全的转换，因为两者都是类似值的并且该类不能有子类。
- **可能会将没有类继承的抽象类更改为接口。**此转换保持相同的约定，但尚未完全实现（在测试中标记为 TODO）。

### 结构

**无法向结构添加任何字段。**结构是具有固定内存布局的不可变值类型。添加字段会更改此布局并破坏二进制兼容性。

**不能在类和结构之间进行更改。**这些是具有不同语义的根本不同的类型 - 类是引用，结构是值。

### 字段和数据成员

**添加字段：**

- **可以将具有默认值的字段添加到类中。**具有默认值的新字段不会破坏现有的构造代码，因为默认值是自动使用的。
- **无法将没有默认值的字段添加到类中。** 构造实例的现有代码会中断，因为它不为新字段提供值。
- **无法向结构添加任何字段。**结构具有固定的内存布局，添加字段会破坏二进制兼容性。

**删除字段：**

- **无法从类或结构中删除字段。** 所有已发布的字段必须保持可用，因为代码可能会引用它们。

**字段可变性：**

- **可以将最终实例字段更改为非最终实例字段。** 允许以前禁止的突变是一种安全的功能扩展。
- **无法将非最终实例字段更改为最终字段。** 一旦代码依赖于能够改变字段，使其不可变就会破坏该代码。

**字段类型更改：**

对于**最终或非实例数据**（无法覆盖的字段）：
- **可以缩小类型（使其更具体）。** 例如，从 `any` 更改为 `int` 加强了对该字段保存的值的保证。
- **无法扩大类型（使其更通用）。** 例如，从 `int` 更改为 `any` 违反了调用者可以读取特定类型的保证。

对于**非最终实例数据**（可以在子类中覆盖的字段）：
- **不能缩小或扩大类型。** 这些必须保持其确切类型，以防止破坏子类或调用代码中的覆盖。

**默认初始化器：**

- **可以向类字段添加默认初始值设定项。**这使得构造更容易，而不会破坏现有代码。
- **无法从类字段中删除默认初始值设定项。** 删除默认值会破坏依赖它的构造代码。**覆盖：**

- **可以向字段添加覆盖。** 允许在子类中提供更具体的实现。
- **如果覆盖没有缩小类型，则可以删除它。**如果覆盖保持相同的类型，则删除它是安全的。

### 函数和方法

函数签名更改遵循不同的规则，具体取决于函数是 **final/non-instance**（无法重写）还是 **non-final 实例方法**（可以重写）。这些规则反映了类型安全的基本原则：函数可以在接受的内容（逆变）方面变得更加灵活，在返回的内容（协变）方面变得更加具体，但前提是不涉及重写。

**过载管理：**

- **无法删除函数重载。** 每个已发布的函数签名必须保持可用，因为代码可能会调用它。
- **函数重载通过签名进行匹配。** 在检查兼容性时，会比较具有相同参数的函数以确保兼容的类型和效果。

**返回类型（协方差）：**

对于 **final 或非实例函数**（模块函数、静态方法、final 方法）：
- **可以缩小返回类型（使其更具体）。**从 `any` 更改为 `int` 意味着该函数现在保证更具体的返回值，这对于调用者来说始终是安全的。
- **无法扩大返回类型（使其更通用）。** 从 `int` 更改为 `any` 意味着该函数可能返回不同的类型，从而破坏需要整数的代码。

对于**非最终实例方法**（可重写方法）：
- **不能缩小或扩大返回类型。** 这些必须保持其确切的返回类型以确保子类覆盖保持兼容。

**参数类型（逆变）：**

对于**最终或非实例函数**：
- **可以扩大参数类型（使其更通用）。** 从 `int` 更改为 `any` 意味着该函数接受更多输入，这永远不会破坏现有的整数调用。
- **无法缩小参数类型（使其更具体）。** 从 `any` 更改为 `int` 意味着该函数拒绝一些以前有效的参数。
- **可以放宽类型参数约束。**从 `t:subtype(comparable)` 更改为 `t:type` 允许更多类型参数。
- **无法加强类型参数约束。** 反向更改会限制有效类型参数。

对于**非最终实例方法**：
- **不能缩小或扩大参数类型。** 这些必须保持其确切的参数类型，以确保子类覆盖和调用保持兼容。

**可选参数：**

- **可以添加带有默认值的可选命名参数。**这不会破坏现有调用，因为参数是可选的。
- **可以更改可选参数的默认值。** 新调用者获得新的默认值，而现有编译的代码继续使用其编译时使用的值。

**效果（协方差）：**

对于**最终或非实例函数**：
- **可以缩小效果（减少）。** 从 `<transacts>` 更改为 `<reads>` 到 `<computes>` 使函数更加纯净，这始终是安全的，因为期望更多效果的代码可以处理更少的效果。
- **无法扩大效果（增加）。** 从 `<computes>` 更改为 `<reads>` 违反了该功能具有有限效果的合同。

对于**非最终实例方法**：
- **不能缩小或扩大效果。** 这些必须保持其准确的效果，以确保覆盖正确工作。

**可调用表单之间的转换：**

- **无法在普通函数和构造函数之间进行转换。** 这些是根本不同的可调用实体，具有不同的调用约定。
- **无法在函数和参数类型之间进行转换。**函数不能成为类型参数，反之亦然。

**函数体：**- **无法更改透明函数的主体。** 调用者的验证可能依赖于透明函数的函数主体，因此更改可能会破坏调用者。
- **如果没有 `<reads>` 效果，则无法更改不透明函数的主体。** 没有 `<reads>` 的函数可保证引用透明度 - 相同的输入始终产生相同的输出。尽管主体对调用者不可见，但编译器无法验证修改后的主体是否为所有输入保留了此映射，因此它保守地禁止主体更改以确保 `NonReadsFunction() = NonReadsFunction()` 跨版本保留。
- **可以更改具有 `<reads>` 效果的不透明函数体。** 可以通过 `<reads>` 效果观察代码演变。

**理解方差：**

这些规则中的不对称性反映了类型理论中的**方差**：
- **参数是逆变的**：接受更通用的类型（加宽）是安全的
- **返回是协变的**：返回更具体的类型（缩小范围）是安全的
- **效果是协变的**：影响较少是安全的

这些规则仅适用于最终/非实例函数，因为可重写方法必须维护精确的签名以保留里氏替换原则——无论调用基方法，任何子类重写都必须是可调用的。

### 访问说明符

**增加可访问性（允许）：**

- **可以增加定义的可访问性。** 将私有定义公开或保护为公共，可以在不破坏现有代码的情况下扩展访问范围。
- **可以使构造函数更易于访问。**允许更多代码构造实例是一种安全的功能扩展。

**降低可访问性（禁止）：**

- **无法降低定义的可访问性。** 将公共定义设置为受保护、内部或私有会破坏所有使用它的外部代码。将受保护的定义设置为私有会破坏子类访问。
- **不能降低构造函数的可访问性。** 公共的类构造函数不能变为私有或受保护。

### 可持久化类型

可持久化类型需要更严格的兼容性规则，因为它们定义了必须保持无限期可加载的已保存玩家数据的格式。对可持久化类型的更改可能会损坏或丢失已保存的数据。

**可持久化属性更改：**

- **发布后无法添加 `<persistable>` 属性。** 使类型可持久化会更改其序列化行为并施加新的限制。
- **无法删除 `<persistable>` 属性。** 保存的数据取决于这些类型的持久性格式。

**可持久化类字段：**

- **可以将具有默认值的字段添加到可持久化类。** 没有新字段的已保存数据将使用默认值成功加载。
- **无法将没有默认值的字段添加到可持久化类。**旧保存的数据不会有这些字段的值。
- **无法从可持久化类中删除任何字段。** 保存的数据可能包含这些字段值并且必须能够加载它们。

**可持久化结构字段：**

- **无法将任何字段添加到可持久化结构。**结构具有固定布局，并且保存的数据需要精确的结构。
- **无法从可持久化结构中删除任何字段。**已保存数据中的所有字段都必须可加载。

**可持久化枚举更改：**

对于**封闭的可持久化枚举**：
- **无法添加枚举器。** Case 语句可能会穷举匹配所有值，并且保存的数据反序列化取决于固定集。
- **无法删除枚举器。** 保存的数据可能包含已删除的值，使其无法加载。

对于 **开放可持久化枚举**：
- **可以添加新的枚举器。** 开放枚举旨在增长，并且持久性系统处理未知值。
- **无法删除枚举器。** 保存的数据可能仍引用已删除的值。

**类型生命周期：**

- **可以添加新的可持久化类型。** 始终允许发布新的可持久化数据类型。
- **一旦发布就无法删除可持久化类型。** 它们必须保持可用于反序列化旧的保存数据。
- **无法更改可持久类型的结构。** 字段类型、继承关系和其他结构更改会破坏反序列化。

**模块/类型别名：**- **可以将模块或类型别名添加到可持久类型。**这提供了引用现有类型而不更改它们的其他方法。
- **可以删除可持久类型的模块或类型别名。** 删除别名不会影响基础类型的持久性。
- **模块别名必须引用相同的路径。**更改目标会破坏使用别名的所有代码。
- **类型别名必须引用相同的类型。** 更改别名类型会破坏使用别名的所有代码。

**非可持久化的更改：**

- **可以自由添加或删除非可持久化类型。** 没有 `<persistable>` 的类型不会影响保存的数据，可以根据需要添加或删除。

**持久变量：**

Verse 支持跨会话维护值的持久变量（模块范围内的 `var` 声明）：

- **可以添加新的持久变量。**新变量使用其默认值进行初始化。
- **无法删除持久变量。** Metaverse 期望这些变量持久存在。
- **无法更改持久变量类型。** 保存的值必须与预期类型匹配。
- **非持久变量可以自由更改。** 局部变量或实例变量不持久并且可以修改。

### 参数类型

参数类型（带有类型参数的泛型类型）还有其他兼容性注意事项：

**类型参数约束：**

- **可以扩大参数类型域中的类型参数约束。**使约束更加宽松（例如，从 `t:subtype(comparable)` 到 `t:type`）允许更多类型参数。
- **无法缩小类型参数约束。** 限制有效类型参数会破坏使用参数类型的现有代码。
- **在检查参数类型内的函数时，类型参数被视为严格的。**这确保了通用上下文中的类型安全。

**参数与非参数：**

- **无法在参数形式和非参数形式之间进行转换。** 参数类型不能丢失其类型参数，而常规类型不能获得它们。
- **无法在函数和参数类型之间进行转换。** 这些是根本不同的构造。

**方差：**

- **方差是从使用中推断出来的，而不是声明的。**类型参数的使用方式（在输入位置、输出位置或两者中）决定了它们的方差。
- **无法更改推断的方差。** 一旦发布类型参数的使用模式，它就会建立必须维护的方差契约。

### 总结

该目录代表了 Verse 强制执行的核心兼容性保证。虽然这些限制可能看起来很广泛，但它们确保已发布的代码仍然是 Metaverse 生态系统的稳定基础。

要记住的关键原则：

1. **添加通常是安全的**：新的可选字段、新的重载、开放枚举中的新枚举器
2. **通常禁止删除**：删除公共定义会破坏依赖代码
3. **缩小范围通常是安全的**：更具体的返回类型，更少的影响
4. **加宽是选择性安全的**：更通用的参数类型（逆变）
5. **最终/非实例成员更加灵活**：可以进化类型和效果
6. **非最终实例成员是严格的**：他们必须维护准确的签名
7. **持久类型是最严格的**：保存的数据施加永久约束

在这些限制下工作的关键是深思熟虑的初始设计——从一开始就选择正确的可见性、最终性、效果和类型属性。在做出这些不可逆转的决定时，请考虑未来的演变需求。

## 长寿的设计理念

创建在较长时间范围内仍然可行的代码需要采用不同的软件设计方法。开发人员必须超越眼前的功能，考虑他们的代码将如何发展并与未来系统交互。这种前瞻性的方法影响着开发的各个方面，从最初的设计到持续的维护。当使用持久类型时，模式规划变得至关重要。由于这些内容在发布后无法更改，因此开发人员不仅必须仔细考虑当前的需求，还要仔细考虑未来潜在的需求。这可能意味着包含并非立即必需的可选字段，或者在未来可能进行扩展时选择开放枚举而不是封闭枚举。做出错误决策的成本（被锁定在不灵活的模式中）会鼓励彻底的前期设计。

效果规范提供了一个有趣的权衡。虽然 Verse 允许并且有时鼓励过度指定效果，将函数标记为具有当前未使用的效果，但这为将来的实现更改提供了灵活性。标记为 `<reads>` 的函数稍后可以优化为 `<computes>`，而不会破坏兼容性，但反之则不然。这种不对称性鼓励保守的效果声明，为未来的修改留出空间。

开放式和封闭式结构之间的选择代表了另一个长期决策。开放枚举允许在发布后添加新值，以防止详尽的模式匹配为代价提供可扩展性。封闭枚举提供了相反的权衡。了解何时灵活性或完整性更有价值需要考虑代码不仅在今天，而且在未来几年将如何使用。
